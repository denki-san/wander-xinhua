# Codex × Blender：可验证的 3D 协作工作流

更新时间：2026-07-19

## 结论先行

GPT-5.6 Sol 对 Blender 的真正强项不是替代 3D 美术师用鼠标逐点建模，而是把一轮本来分散的工作变成可重复的工程闭环：理解参考图和目标、编写或修改 `bpy` 脚本、驱动 Blender 生成 `.blend`/`.glb`、渲染预览、读取报错与产物，再按验收标准迭代。

因此，最适合它的任务是：程序化体块、场景搭建、材质与灯光的规则化调整、批量导出、格式/节点/贴图合规检查，以及从参考照片到可编辑 blockout 的第一版。复杂角色雕刻、审美取舍、未被明确描述的背面结构、拓扑质量和最终镜头感，仍需要人给约束和验收。

> 这里的“操控 Blender”必须拆开理解：Codex 能生成和维护 Blender Python；但是否能直接点 UI，取决于本机是否另行提供 Computer Use、可控浏览器、Blender 插件或 MCP 桥接。没有桥接时，可靠默认路径是 Blender 命令行后台执行，而不是假定模型拥有 GUI 控制权。

## 能力边界

| 工作环节 | Sol 可以高效承担 | 人必须保留的判断 |
| --- | --- | --- |
| 需求转译 | 将照片、尺寸、风格、目标平台整理成可执行建模规格 | 目标是否真实、是否允许风格化、优先级 |
| 建模 | 写确定性 `bpy` 生成器，搭体块、重复构件、相机、灯光、导出设置 | 建筑背面/遮挡细节的取舍，艺术完成度 |
| 执行 | 通过 `blender --background --python` 运行脚本，读日志并修复明确错误 | 首次执行授权、外部资源与插件信任 |
| 验收 | 渲染固定机位预览，检查 GLB 结构、文件存在性、节点/纹理策略 | 是否“像”、是否可用于最终产品、版权与事实边界 |
| 沉淀 | 将参数、证据、脚本、预览和复盘写成可搜索资产 | 哪些经验应升格为团队规范 |

## 推荐的最小闭环

```text
参考图 / 尺寸 / 风格简报
          ↓
可审阅建模规格（事实、推断、未知项分开）
          ↓
参数化 bpy 生成脚本 ──→ Blender 后台执行
          ↓                        ↓
 .blend（可编辑） + .glb（运行时） + 固定机位 PNG（可见验收）
          ↓
结构审计 + 视觉审阅 + Three.js/目标引擎复验
          ↓
证据清单 + 决策日志 + 可复用组件/提示词
```

这条链路的关键不是一次生成“完美模型”，而是每轮都有可比较的产物。程序化脚本是可回退的模型源代码；`.blend` 是人工继续编辑的源资产；预览和审计报告才是对“这次改动是否真的变好”的证据。

## 在本项目中的落点

`building-evidence-lab` 已经具备这条链路的骨架：

- `research/references/`：原始参考证据，只用于研究；
- `research/source/*.py`：确定性的 Blender 生成器；
- `research/source/*.blend`：可编辑源资产；
- `research/previews/test_*_preview.png`：固定机位的视觉验收；
- `public/models/*.glb`：网页运行时资产；
- `research/poi-evidence-manifest.json`：照片、来源、源文件、预览的关联证据。

现有 `xinhua-wonder-workflow.md` 已明确：单张照片应产出的是 **evidence-conditioned procedural blockout**，不是摄影测量真值；Blender 中预览通过后，还要在 Three.js 中检查比例、地面接触、遮挡、相机和性能。

## 建议的执行方式

### 1. 默认：后台脚本模式

把每个目标写成明确的 Python 生成器，由 Blender 执行。官方命令行支持 `--background`、`--python`，以及将 Python 异常转换为非零退出码的选项；这特别适合让 Codex 读取日志、修复脚本并重跑。

```bash
BLENDER_BIN="/Applications/Blender.app/Contents/MacOS/Blender"
"$BLENDER_BIN" --background --python-exit-code 1 \
  --python research/source/create_example.py
```

实际路径与 Blender 版本应在运行前探测；不要把这段示例当成“本机已安装 Blender”的证明。

脚本应遵循四个约束：

1. 所有比例、材质色、相机和导出路径集中为参数；
2. 每次输出 `.blend`、`.glb` 与固定机位 PNG，避免只留临时视口结果；
3. 失败时让进程明确退出，并在日志里打印产物路径和关键统计；
4. 不从未知 `.blend` 自动运行脚本；Blender 的 Python 有完整本机权限，下载来的文件应默认不信任。

### 2. 需要交互时：把 GUI 作为最后一公里

有 UI 控制桥接时，适合做这些短动作：打开文件、观察渲染、选择对象、拍视口截图、确认人工调整。不要把核心几何逻辑埋在不可复现的鼠标操作里；人工修改后应反向写回参数或生成器。

### 3. 不推荐：一次性大提示词直接“造完整场景”

这类任务很容易看起来有产物，实际缺少尺度、坐标、证据、导出约束与验收方式。正确拆法是：先 blockout，再身份构件，再材质/灯光，最后目标引擎复验。

## 给 Codex 的任务模板

```text
目标：为 <资产名> 生成可编辑 Blender 资产，并导出 <GLB/FBX/...>。

输入证据：
- 参考图：<绝对路径或 URL；说明版权与是否可发给云端>
- 已知尺寸/轮廓：<数值与来源>
- 必须保留的识别点：<至少 3 项>
- 风格：<写实/低多边形/新华 Wonder...>

约束：
- 未被证据支持的部分标为推断，不伪装成事实；
- 以 bpy 脚本实现，所有可调数值集中为参数；
- 保存 <name>.blend，导出 <name>.glb，渲染两张固定机位预览；
- 不嵌入未授权照片或贴图；
- 不覆盖已有资产，先写新版本目录。

验收：
1. Blender 后台脚本无异常；
2. 预览图可读，且能看到三个识别点；
3. 导出文件存在，通过结构审计；
4. 在目标引擎复核比例、坐标、遮挡与性能；
5. 输出变更摘要、未知项和下一轮建议。
```

## 知识如何真正沉淀

每个资产应至少留下这六类内容，才能从“一次成功的对话”变成可复用能力：

1. **Brief**：目标、平台、预算、风格与验收条件；
2. **Evidence manifest**：参考来源、许可、使用范围、已知尺寸与置信度；
3. **Generator**：参数化 `bpy` 脚本，作为可审查、可回退的源代码；
4. **Editable source**：与脚本版本匹配的 `.blend`；
5. **Validation**：固定机位预览、GLB 审计、目标引擎截图/测试；
6. **Decision log**：本轮推断、没有解决的缺口、为何采用某种比例/材质/拓扑。

跨资产重复出现的结论，再升格为 `research/` 中的 Playbook，例如“玻璃幕墙的透明度与排序规则”“屋檐的最小可读厚度”“GLB 不应嵌入参考照片”。不要把单一资产的偶然调参直接写成通用规则。
