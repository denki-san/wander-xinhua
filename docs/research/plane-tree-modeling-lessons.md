# Plane Tree Modeling Lessons

## 结论

梧桐树这轮最有价值的经验，不是某个 Blender 参数，而是把“看起来像”拆成可验证的证据、结构、导出和运行时问题。孙科别墅沿用同一套方法，但把树的枝干层级替换为建筑的体块与立面层级。

## 1. 先定义身份，再追求细节

梧桐树真正可识别的不是叶片数量，而是高位多向分叉、可读的主次枝层级、不规则树冠空隙、斑驳剥落树皮、五裂叶与少量悬挂果球。生成器先完成连续树干和主枝，再挂二三级枝、叶片、树皮补丁和果球，避免用“一根圆柱加一个球冠”冒充树种。

迁移到孙科别墅：先锁定“低左翼 + 中央双层拱廊 + 右侧圆角塔楼 + 错落红瓦屋顶”，再添加窗框、瓦脊和栏杆。只换墙色和屋顶色不能形成独立地标。

## 2. 参考图必须分工，不能混成一张想象图

梧桐树把全树照、街道树阵、树皮细节和风格图分开使用；风格图只决定低多边形语言，不被当成植物事实。每个结论都能回到本地图片和来源元数据。

迁移到孙科别墅：花园正立面负责整体比例，右前斜视负责塔楼与纵深，北入口负责门廊和后侧体块。不可见立面明确标记为推断，不把园区其他历史建筑误作同一主体。

## 3. 有机形态需要确定性，也需要局部随机隔离

`create_plane_tree.py` 使用固定 `SEED` 保证每次重建一致；板根又使用独立的 `random.Random(SEED + 81)`，使根部局部迭代不会意外改变已经确认的树冠。确定性不是让造型机械，而是让每次视觉比较只包含有意修改。

迁移到孙科别墅：建筑本身不使用随机体块；瓦片错位、墙面色差和景观植物若需要变化，也必须使用独立固定种子，不能影响主体几何。

## 4. 连续结构优于零散零件

树干和枝条改用连续管状网格，避免每一节锥体在接缝处产生阴影断裂；板根从根颈向地面展宽并逐渐收尖，解决了“树干插在地面上”的漂浮感。树根也证明了一个问题：即使代码逻辑正确，固定机位预览仍可能暴露根颈过厚、板根过尖或分布过均匀。

迁移到孙科别墅：屋顶、墙体、塔楼和门廊要在转角处形成连续轮廓；窗套和拱券不能只是贴在一块大盒子上的平面符号。canonical、斜视和背侧机位必须同时检查。

## 5. 材质用小色盘表达身份，不依赖照片纹理

梧桐 GLB 没有嵌入图片，使用少量树皮、剥落斑块、叶色和果球材质表达层次。当前 GLB 审计为 `7` 个节点、`7` 个网格、`7` 个材质、`0` 张图片，说明参考照片和运行时资产边界保持清楚。

迁移到孙科别墅：拉毛墙、红瓦、暖石材、深色窗框、玻璃和黑铁构成最多八种材质；照片不进入 GLB，粗糙墙面主要靠轮廓、光照和有限的几何层次表达。

## 6. 视觉正确以后再优化节点

树的枝叶在建模阶段保持可理解的对象结构，确认形态后再按材质合并；这样既保留可编辑 `.blend`，又把运行时节点控制在预算内。过早合并会让局部修正困难，过晚合并会把重复对象成本带进浏览器。

迁移到孙科别墅：体块、窗门、屋顶、场地分别建造和检查，最终合并为一到两个静态网格，保留材质槽语义。优化不能删除塔楼、三联尖券、阳台或北门廊等身份构件。

## 7. 专用观察模式比“凭感觉好看”更可靠

梧桐 Lab 提供转台、手动观察、隐藏树叶和查看枝干结构等模式。隐藏树叶后仍能辨认树种，才说明树冠不是靠叶片密度掩盖骨架问题。

迁移到孙科别墅：固定的花园正面、右前斜视和北入口三机位，分别检查正面身份、塔楼纵深和背侧完整性；运行时再复现 canonical 方向，避免 Blender 里好看、放进园区后朝向或比例错误。

## 8. Blender、GLB、Three.js 是三套不同的完成条件

梧桐源文件、导出文件和预览都存在，GLB 结构审计也通过，但这不自动证明运行时相机、触摸交互、碰撞、地面接触和性能正确。历史上 Blender 可执行路径失效还曾让源代码改进无法重建成产物，说明工具链状态也必须验证。

迁移到孙科别墅：生成器成功、GLB 审计通过和专用 `/?start=sunke` 实页验收必须分别记录；任何一层缺失都不能称为模型完成。

## 9. 重复环境资产需要“结构变体”，不是随机缩放

梧桐树作为 hero parent，只解决一棵树的结构质量；真正上街需要至少三种不同树干倾斜、主叉角度、冠幅、密度和修剪轮廓，并通过实例化或合批控制成本。

迁移到孙科别墅：别墅是唯一地标，不做随机变体；周边树篱和植物才采用低成本确定性变体，且不得遮挡入口、塔楼和 canonical 视线。
