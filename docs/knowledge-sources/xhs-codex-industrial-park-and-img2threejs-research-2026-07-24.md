# 两条小红书 3D 建模笔记：产业园自动建模与 img2threejs

研究日期：2026-07-24（Asia/Shanghai）
用途：Three.js / Blender / 建筑建模学习资料；不作为任何真实 POI 的建模许可或照片证据。

## 结论先行

两条笔记代表互补的路线，而不是同一件事：

1. **产业园案例**适合被理解为“AI 先完成建筑体块、构件层次与场地初稿，再在 Rhino 等 DCC 中人工核验和精修”的演示。它没有公开输入数据、代码、模型格式、误差或基准，不能据此相信“一张图、几分钟、零修补”可复用于真实建筑。
2. **img2threejs**是开源的“参考图 → 可编辑 Three.js 代码资产”工作流；它并不直接输出已恢复的网格，也不是 photogrammetry。它适合小型、硬表面、可拆件的网页道具，价值在质量门、结构化 spec 和可追踪的代码资产，而非单图还原真实建筑。

对于 Wander Xinhua：前者只能作为 **massing / blockout 到人工精修的流程参考**；后者可以试用于非核心街道道具，不能替代重点历史建筑的多视角研究、Blender 建模和真实运行时验收。

### 可吸收的工作流判断

可复用的是两阶段分工，而非“自动完成”的承诺：

```text
受约束的建筑输入
  → AI / 规则系统产出 massing 与可审查构件初稿
  → 剖面、图层、尺寸、关键立面核验
  → Rhino / Blender 人工精修与导出
  → Three.js 真实场景、碰撞和性能验收
```

当目标是 Wander Xinhua 的真实建筑时，至少仍需要 canonical、侧向、入口/身份细节三个视角的照片证据；不可见的背面、室内和结构尺寸必须保持 `unknown`，不能因自动初稿而转成事实。

### 笔记直接陈述

笔记明确指向仓库 `hoainho/img2threejs`，并称它能把照片转成“完全交互式 3D 模型”，不需要 mesh 文件、下载模型或调用模型生成 API，而是使用 Three.js。

### 上游仓库核验

- 官方仓库：<https://github.com/hoainho/img2threejs>
- 本次核验提交：`c9077d5ecce834f6802d6742b4a5b2c682d6279d`（2026-07-24 05:29:27 UTC）；该次提交是 README 文档更新。
- 当前 `SKILL.md` frontmatter：版本 `1.3.0`、许可证 Apache-2.0。
- 官方定义：根据参考图，以**纯代码、程序化、分阶段质量门**重建 Three.js 模型；明确不等于 photogrammetry、mesh extraction 或下载外部资产。

它的目标输出是可组织、可编辑的 `THREE.Group` / TypeScript factory：结构层级、pivot、socket、collider 等运行时语义可以存在于代码中。这里的“animation-ready”主要意味着可独立运动的组件与运行时层级；不能自动等同为已有标准角色骨骼、蒙皮、morph target 或 glTF 交付。

### 可确认的工程价值

官方流程要求先做图像可用性与质量合同，再依次通过：

```text
blockout → structural → form → material → lighting → interaction → optimization
```

每一阶段应有实际渲染、参考/渲染对照和明确的继续或返工决定。这与本项目的模型 Brief、视角覆盖矩阵、三联对照和 `?start=` 运行时验收可直接对接。

### 单张照片的硬边界

单图只提供投影视图；遮挡面、背面、真实尺寸和材料物理属性不能由它证明。对于玻璃幕墙建筑、树木、人物、布料、镜面、多遮挡街景或需要照片级身份还原的 POI，不能把本路线作为唯一建模方法。
